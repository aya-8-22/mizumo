# 名前：ミズモ（水＋メモ）

## 1.サービス概要
高齢者が、毎日の水分補給を習慣化するためのアプリ
飲水記録を簡単に入力でき、カレンダーで達成状況を視覚的に確認できる
メール通知機能により、飲水のタイミングを忘れずにリマインドする

## 2.このアイデアはどこから生まれたか
### 2-1. きっかけとなった体験・感情
- いつ・どんな状況で：
医療機関で働いていた時、夏場になると発熱で搬送される高齢者が急増する状況を何度も目にしてきた
特に印象的だったのは、高齢の患者様がよく「そういえば、あんまり水分とっていなかった」と話していた

- どんな気持ちになったか：
発熱があると、コロナウイルスやインフルエンザの可能性を考慮し、隔離や感染対策を行う必要がある
検査で陰性が確認されるまで、感染対策を継続する必要がある
その時、私は「もし水分補給を習慣化できていれば、この患者様は熱中症にならず、医療機関の逼迫も予防できたのではないか」と強く感じた
2025年（令和7年）の統計では、65歳以上の熱中症による救急搬送は全体の約6割（総務省）に及んでおり、この問題は個人の問題ではなく、医療現場全体の課題だと実感した

### 2-2. なぜそれが「気になった」のか
- 自分の性格、過去の経験
医療現場で働いていた経験から、「予防できる病気で医療機関が逼迫する」という状況に強い違和感を感じていた
高齢者は体温調節機能の低下、認知機能や判断力の低下により、水分摂取を忘れやすい傾向にある
しかし、「水分補給を忘れないようにする」という習慣化さえできれば、熱中症は予防できると考えた

- 大切にしている価値観：
私は「予防できる病気は予防すべき」という考えを大切にしている
特にコロナ禍以降、医療機関では感染対策の負担が増しており、「予防できる病気を減らすことで、医療現場の負担を軽減したい」という思いが強くあった
また、医療現場で働いていた時に、「もっと早く対策できていれば」と感じる場面を何度も経験したことが、このアプリを作りたいと思った理由である

3. 課題の整理（表面的になっていないか）
3-1. 表に見えている困りごと
どんな生活場面で困るのか：
  - 朝起きてから夕方まで、一度も水を飲まずに過ごしてしまうことがある
  - 高齢者は「喉が渇いたら飲めばいい」と思っているが、喉の渇きを感じにくいため、気づいたときには頭痛やめまいなどの脱水症状が出ている
  - 他者が「水飲んだ？」と声掛けしても、「さっき飲んだ」と答えるが、実際には飲んでいないこともある

何が不便なのか：
  - 水分補給を記録するアプリはあるが、記録を忘れてしまう
  - 記録を忘れると、達成状況が正しく反映されず、アプリの価値が下がる
  - 他者が毎回声掛けするのは負担が大きい

なぜ困るのか：
  - 水分補給を忘れると、脱水症や熱中症のリスクが高まる
  - 特に夏場は、気づかないうちに脱水状態になり、救急搬送されるケースも多い
  - 高齢者は喉の渇きを感じにくいため、意識的に水分補給をする必要がある
  - 医療機関の逼迫にもつながる

なぜWebアプリで解決する必要があるのか：
  - メール通知により、定期的に水分補給を思い出せる
  - 記録が自動的に残り、カレンダーで達成状況を視覚的に確認できる
  - 家族が遠方にいても、達成状況を共有できる(将来的な展開)
  - スマートフォンやパソコンがあれば、どこでも使える

なぜ「声掛け」ではなく「メール通知」なのか：
- 背景:独居高齢者の増加
  - 2025年（令和7年）時点で、65歳以上の独居の割合は女性18.3％、男性25.4％（内閣府調査）となっている
  - 2050年（令和32年）には、女性29.3％、男性26.1％に達すると予測される
  - 今後も独居の高齢者は増加する可能性が高い
  - そのため、他者の声掛けに依存せず、個人で解決できる仕組みが必要である

- 「声掛け」と「メール通知」の違い：
| 比較項目 | 声掛け | メール通知（このアプリ） |
|---------|-------------------|---------------------|
| 継続性 | 家族の負担が大きく、継続が難しい | 自動化されているため、継続しやすい |
| タイミング | 家族の都合に左右される | 決まった時間に確実に届く |
| 記録の正確性 | 口頭のため、記録が残らない | アプリで記録が残り、振り返りができる |
| 心理的負担 | 「また言われた」という抵抗感がある | 機械的な通知のため、抵抗感が少ない |
| 達成感 | 達成状況が可視化されない | カレンダーで達成状況を可視化できる |
| 家族の負担 | 毎日声をかけ続ける必要がある | 家族の負担がなくなる |

なぜ「声掛け」では曖昧になるのか:  
  - 家族が「水飲んだ?」と聞いても、高齢者は「さっき飲んだ」と答えてしまうことがある
  - これは、記憶が曖昧であったり、家族を安心させたいという気持ちから、曖昧に答えてしまう
  - 記録が残らないため、本人も家族も「実際に飲んだかどうか」を確認できない
  
なぜ「メール通知」なら曖昧さが減るのか:
  - メール通知は「行動のきっかけ」を提供する
    - 「水を飲む」という行動を思い出すタイミングを提供する
    - メール内のリンクをタップすることで、記録画面に直接遷移できる
  - 記録が残るため、曖昧さがなくなる
    - 「今日は何回飲んだか」がカレンダーで一目でわかる
    - 「さっき飲んだ」という曖昧な回答ではなく、記録として残る
  - ワンタップで記録できるため、記録のハードルが低い
    - 既存アプリのように、飲料の種類や量を選ぶ必要がない
    - メールのリンクをタップするだけで、記録が完了する

MVP版で検証したいこと:
  - メール通知が届くことで、水分補給を思い出せるか
  - メールのリンクから記録画面にスムーズに遷移できるか
  - 固定時刻の通知でも習慣化に役立つか
  - 記録が残ることで、「今日は飲んだ」が明確になるか
  - ログイン画面を経由する操作が、高齢者にとって負担になるか確認する

3-2. 本当に解決したい課題は何か
- なぜ水分補給を忘れるのか？
  - 加齢により、喉の渇きを感じにくくなりやすくなる

- なぜ喉の渇きを感じにくいのか？
  - 加齢により、体の水分量が減少し、喉の渇きを感じる機能が低下しているからである

- なぜそれが問題なのか？
  - 水分不足が続くと、脱水症や熱中症などのリスクが高まり、健康を損なうからである

- 本当に解決したい課題
  - 高齢者が、喉の渇きを感じなくても、定期的に水分補給を思い出し、習慣化できるようにする

## 4. 想定ユーザーについて
### 4-1. 想定しているユーザー
年齢設定：
- 年齢：
  - 65歳〜75歳の比較的元気な高齢者
- 理由：
  - 熱中症による救急搬送の6割が65歳以上であり、この年代にアプローチすることで、熱中症予防に大きな効果が期待できる
  - 65歳〜75歳はスマートフォンやメールを日常的に使っている方が多いため、アプリの導入ハードルが低いと考えた
  - 今後、独居高齢者の割合が増加すると予測されている
  - 独居高齢者は家族の声掛けに依存できないため、自己完結で使えるアプリが必要である
  - 軽度認知障害（MCI）までは携帯を利用できる可能性があるため、この層も対象に含めたいと考えた

ITリテラシー：
- スマートフォンやメールを日常的に使える方
- メールのリンクをタップして、簡単な操作ができる方

健康状態：
- 飲水制限がない方
- 軽度認知障害（MCI）までの方
- 比較的元気で、日常生活を自立して送れる方

生活環境：
- 独居または日中独居の方
- 家族の声掛けに依存できない環境にある方

ターゲットユーザーの明確化：
- 最初に登録するのは誰か：
  高齢者本人が登録する
  家族がサポートすることもありますが、基本的には高齢者本人が自分で登録できることを前提とする
- 毎日使うのは誰か：
  高齢者本人が毎日使う
  メール通知を受け取り、ワンタップで記録を行う
- 継続を支えるのは誰か：
  高齢者本人がメール通知とカレンダーの達成感によって継続できる仕組みにする
  家族は「達成状況を見守る」程度の関わり方を想定しており、毎日の声掛けやサポートは不要である

なぜ高齢者本人が自力で継続できるのか：
  - メール通知により、定期的に水分補給を思い出せる
  - ワンタップで記録できるため、操作が簡単
  - カレンダーで達成状況を確認できるため、達成感が得られる
  - 家族の声掛けに依存せず、アプリの仕組みで習慣化できる

環境設定：
- 環境：
  - メール通知を採用する
- 理由：
  - 高齢者の中には、LINEやスマホアプリに不慣れな方も多くいる
  　一方で、メールはフィーチャーフォン（ガラケー）時代から使い慣れている方が多く、スマホに機種変更した後もメールアプリは標準でインストールされているため、導入のハードルが低いと考えた
  　ただし、将来的にはLINE通知にも対応することで、より幅広く使ってもらえるようにしたいと考えている

将来的な展開：
  - まずは元気な高齢者で実績を作り、その後医療機関や飲水制限のある高齢者にも展開したいと考えている
  - 医療機関や介護施設での導入には、医師の監修や飲水制限への対応が必要になるため、まずは個人ユーザーでの実績を作ることを優先する

### 4-2. 自分とユーザーの距離
  - 医療現場での経験から、「水分補給を忘れていた」と話す高齢者を何度も見てきた
  　また、自分の家族にも高齢者がいるため、「将来的に身近な人にも使ってもらいたい」という思いが強くある

まず最初に使ってくれそうな人は誰か？

その人は、どんな場面でこのアプリを思い出すか？

### 4-4. ユーザーがサービスを導入して利用するまでのイメージ（導入・継続の流れ）
どんな場面・困りごとがきっかけで、このサービスを知る・思い出すか
  - 家族や友人が「水分補給を忘れないようにするアプリがあるよ」と勧めてくれたとき、または自分自身が「最近水を飲むのを忘れがち」と感じた時に思い出す

最初の一歩（登録・導入）で、ユーザーは何をする必要があるか
  - メールアドレスと体重を登録するだけで、複雑な設定は不要で、すぐに使い始められる

利用を続ける理由は何か？（習慣・必要性・価値）
習慣化しやすい仕組み：
  - メールでリマインドが届くので、自然と水分補給を思い出せる
記録が簡単：
  - メール内のリンクをタップするだけで、ワンタップで記録完了する
  - アプリを開く手間がないため、高齢者でも簡単に記録できる
達成感が得られる：
  - カレンダーで達成状況を確認できるので、「今日も達成できた」という小さな達成感が継続のモチベーションになる

離脱ポイント（どんなときに使わなくなるか）
メール通知が煩わしく感じたとき： 
  - 対策として、通知頻度をカスタマイズできる機能を追加することで、ユーザーに合わせた通知設定が可能になる
記録を忘れてしまったとき：
  - 対策として、「記録を忘れても大丈夫」というメッセージを表示し、プレッシャーを感じさせないようにする
達成できない日が続いたとき： 
  - 対策として、今日は達成できなくても、明日からまた頑張りましょう」といった励ましのメッセージを表示する

### 5-2. それでも自分が作りたい理由
既存サービスがある中で、
  - なぜ満足できなかったのか
  - どこに違和感があったのか
1. 若い世代向けが中心
  - 9歳以上の健常者を対象にしたアプリが多く、高齢者への配慮が少ない
2. 操作が複雑で継続しづらい
  - 飲料の種類が20種類以上あり、記録のたびに複数の項目を入力する必要がある
3. 文字が小さく、視認性が低い
  - 高齢者にとって読みにくいデザインが多い傾向にある
4. 達成状況が分かりにくい
  - グラフや棒グラフで表示されるが、目標達成できたかが一目でわかりにくい

### 5-3. 差別化を一文で決める
私のアプリの差別化ポイント：
このアプリは、「①水分摂取をメール通知で思い出す → ②ワンタップで記録 → ③カレンダーで達成感を得る」というサイクルが、習慣化に繋がり、継続しやすい

既存アプリとの違い:
| 項目 | 既存アプリ | ミズモ |
|------|-----------|--------|
| 記録方法 | アプリを開いて、飲料の種類・量を選択 | メール内のリンクをタップするだけ |
| 通知方法 | アプリ内通知(見逃しやすい) | メール通知(見逃しにくい) |
| 達成感 | グラフや数値で表示(分かりにくい) | カレンダーで○表示(分かりやすい) |
| 継続のしやすさ | 記録を忘れると使わなくなる | メール通知で思い出せるので継続しやすい |

## 6. このサービスで提供したい価値
### 6-1. ユーザーの変化
行動や気持ちの変化：
MVP後:
1. メールの通知で水分補給を思い出せる
  - 家族の声掛けに依存せず、定期的に水分補給を思い出せる
  - 「今、水を飲もう」という行動のきっかけを提供する
2. ワンタップで記録できるため、記録が続けやすい
  - メール内のリンクをタップするだけで、記録画面に遷移できる
  - 既存アプリのように、飲料の種類や量を選ぶ必要がない
  - 記録のハードルが低いため、継続しやすい
3. カレンダーで達成状況を視覚的に確認できる
  - 「今日も達成できた」という小さな達成感が継続のモチベーションになる
  - 達成状況を一目で確認できるため、習慣化の進捗を実感できる

その先に期待できる変化:
 - 短期的変化（数週間〜数ヶ月）:
  - 水分補給が習慣化される
    - 「喉が渇いたら飲む」から「時間になったら飲む」という行動変化
    - 1日の水分補給回数が増え、目標水分量を達成できるようになる
 - 中期的変化（数ヶ月〜1年）:
  - 脱水症や熱中症のリスクが軽減される
    - 水分補給が習慣化されることで、脱水状態になりにくくなる
    - 夏場の熱中症リスクが軽減される
 - 長期的変化（1年以上）:
  - 救急搬送の軽減により、医療機関の逼迫を防ぐ
    - 予防できる病気で医療機関が逼迫するのを防ぐ
    - これはアプリ単体で実現できる価値ではなく、多くのユーザーが習慣化した結果として期待できる社会的な価値である

### 6-2. 価値を一文で表す
「このサービスは、水分補給を忘れがちな高齢者が、メールのリマインドとカレンダーの視覚化によって、毎日の水分補給を自然に習慣化できるようになるサービス」

## 7.このアプリで実現すること
### MVP版開発前の事前検証：
1. 認証だけの簡易アプリを用意し、メールのリンクから遷移できるか検証
  - Railsで認証機能だけを持つ簡易アプリを作成する（Deviseを使用）
  - 実際に身近な高齢者にメールを送信し、ログイン画面を経由する操作の負担を検証する
  - 対象者：２名以上
  - 検証期間：３日程度
  - 検証結果をもとに、MVP版の仕様を調整する

### MVPで作る機能（Must：必須）
1. ユーザー登録・ログイン機能（Devise）  
  - ユーザーごとに記録を管理
  - 登録時にメールアドレス、パスワード、体重を登録

2. 体重入力と目標飲水量の自動計算（Rails標準）  
  - 体重を入力すると、目標飲水量が自動で計算（体重 × 30ml）

3. 水分補給記録のCRUD機能（Rails標準）
  - ワンタップ記録（コップ1杯 = 200ml固定）
  - 記録の編集・削除も可能

4. カレンダー表示機能（Simple Calendar）
  - その日の目標飲水量を達成した日に⚪︎が表示される
  - カレンダーで、その日をクリックすると水分補給回数と量を確認できる
  - 一目で習慣化の進捗を確認できる

5. メール通知機能（Action Mailer + Render Cron Jobs）  
  - 1日3回（朝8時、昼12時、夕方17時）にメール通知を送信
  - メール内に記録画面へのリンクを含める
  - リンクをタップすると、ログイン画面に遷移（Deviseの標準機能を使用）
  - ログイン後、記録画面に遷移し、「飲んだ」ボタンをタップで記録完了
　
### MVP完成後に追加する機能（Should：優先）
6. パスワード管理機能の改善
  - パスワードリセット機能を用意
  - 初回登録時に「パスワードを忘れた場合の対処法」を案内する
  - パスワード入力画面に「パスワードを表示する」チェックボックスを追加
  - パスワードの強度チェックを緩和（最低8文字、英数字のみでOK）
  - 大文字小文字の区別をなくす（すべて小文字として扱う）
  - 記号の入力を不要にする
  - パスワード入力時に、入力中の文字が見えるようにする(デフォルトは非表示、チェックボックスで表示切替)
  - パスワード入力欄の下に「入力した文字」を表示する(例:「a」を入力したら「a」と表示)
  - パスワード入力時に、大文字小文字の入力切替方法を案内する(例:「Shiftキーを押しながら入力すると大文字になります」)
  - パスワード入力時に、何を入力したか状態がわかるようにする(例:「8文字入力しました」と表示)

7. 通知時刻のカスタマイズ機能（Sidekiq + Whenever）  
  - ユーザーごとに好きな時間に水分補給リマインドを設定できる
  - 通知回数の調整（1日6~10回）
  - デフォルトの通知時刻を提案（起床時、朝食時、10時頃、昼食時、15時頃、夕食時、入浴前後、就寝前）
  - ユーザーが自由に追加・削除・編集できる

8. 就寝時の達成状況通知（Action Mailer + Sidekiq + Whenever）
  - 就寝前に、水分補給記録（回数、合計水分量、達成率）をメールで通知
  - 達成率に応じてメッセージを変え、ポジティブフィードバックで達成感を得られやすくする
    - 100%達成:「🎉 目標を達成しました!素晴らしいです!」
    - 80%以上:「👍 もう少しで目標達成です!明日も頑張りましょう!」
    - 80%未満:「💪 明日は目標達成を目指しましょう!」
  - メール内のリンクから記録画面に遷移できる
  - リンクがクリックできない場合の対処法を記載する:
    - 「リンクがクリックできない場合は、以下のURLをコピーしてブラウザに貼り付けてください」
  - 記録画面にはカレンダーが表示され、目標達成した日に○が表示される
  - Should版では、トークン認証により、ログイン画面を経由せずに記録画面に遷移できる

9. トークン認証（自動ログイン）
  - リンクには、ユーザーを識別するトークンを含める
  - ユーザーがメールのリンクをタップすると、自動的にログイン状態になり、記録画面に遷移
  - トークンの有効期限は24時間とする
  - トークンは一度しか使用できないようにする（ワンタイムトークン）
  - トークンは推測されにくいランダムな文字列を使用する（Railsのhas_secure_token を使用）
  - ユーザーはログイン画面を経由せずに、ワンタップで記録できる

10. メール配信サービスの導入（SendGrid または Mailgun）
  - メール到達率を向上させるため、専門の配信サービスを導入
  - SPF、DKIM、DMARCの設定を行い、迷惑メールフォルダに振り分けられないようにする

## 8.このアプリの懸念点とその対策
懸念点①：メール通知が迷惑メールフォルダに入ってしまう
何が問題になりそうか：
  - メール通知が迷惑メールフォルダに振り分けられると、ユーザーがリマインドを受け取れない可能性がある
  - 一部のメーラーで、URLがリンクにならないケースがある（特に古い機種やPC）
なぜそう思うか：
  - 定期的に送信されるメールは、スパムと判定されやすい
  - iOSの一部の機種やPC含めて一部のメーラーで、URLがリンクにならないケースがある
そのために考えている対策：
MVP版：
  - ユーザーがメールアドレスを登録する際に、テストメールを送信して受信確認を促す
  - 初回登録時に「メールが届かない場合は迷惑メールフォルダを確認してください」と案内する
  - メール内に「リンクが表示されない場合は、以下のURLをコピーしてブラウザに貼り付けてください」という文言を追加する
    - リンクの下に、URLをテキストで表示する
Should版：
  - SendGridやMailgunなどの信頼性の高いメール配信サービスを使用する
  - 送信元ドメインの認証（SPF、DKIM、DMARC）を設定して信頼性を高める

---

懸念点②：ユーザーが水分補給記録を入力し忘れる
何が問題になりそうか：
  - 水分補給記録を入力しないと、達成状況が正しく反映されず、継続のモチベーションが下がる
なぜそう思うか：
  - 高齢者は、スマホの操作に慣れていないため、記録を忘れやすい
そのために考えている対策：
MVP版:
  - 記録入力を簡単にする（ワンタップで記録完了）
  - メール通知に「記録はこちらから」というリンクを含める
  - 水分摂取量は「コップ1杯 = 200ml」と固定し、選択の手間を省く
Should版：
  - 就寝時間前に「今日の記録を確認しましょう」というリマインドメールを送る
Could版（将来的に追加）：
  - 設定画面で「コップ1杯の量」を選択できるようにする
  - 選択肢：150ml / 200ml / 250ml など
  - 記録時は、設定した量が自動的に記録される

---

懸念点③：目標水分量の計算が実態と合わない可能性
何が問題になりそうか：
  - 体重だけで目標水分量を計算すると、個人の生活習慣や健康状態に合わない可能性がある
なぜそう思うか：
  - 運動量、気温、持病などによって必要な水分量は変わる
そのために考えている対策：
MVP版：
  - 「体重 × 30ml」というシンプルな計算式を採用する
  - 「医師に相談してください」という注意書きを表示する
将来的な展開：
  - 運動量や気温を考慮したカスタマイズ機能を追加する

---

懸念点④：実際に水を飲んでいなくても記録できてしまう
何が問題になりそうか：
  - メール通知のリンクをタップするだけで記録が完了すると、実際に水を飲んでいなくても記録できてしまうので「水分補給を習慣化する」という目的が達成できない
なぜそう思うか：
  - 記録が「実際の行動」を反映しない場合、カレンダーの達成状況が「嘘」になってしまう
  - ユーザー自身も「記録したから飲んだことにしよう」と自己欺瞞に陥る可能性がある
そのために考えている対策：
MVP版
1. シンプルな記録フロー
  - メール通知のリンクをタップすると、ログイン画面に遷移（Deviseの標準機能を使用）
  - ログイン後、記録画面に遷移
  - 記録画面で「飲んだ」ボタンをタップすると記録が完了（ワンタップ）
  - 確認メッセージは表示しない（操作のハードルを下げるため）
2. カレンダーでの達成状況の可視化
  - カレンダーで目標達成した日を○で表示
  - カレンダーでは、その日をクリックすると飲水回数と量を確認できるようにする
  - 達成状況を可視化することで、ユーザー自身が「本当に飲んだか」を意識できる
3. 継続性を重視した設計
  - 記録のハードルを下げるため、ワンタップで記録完了できる仕組みを維持
  - 高齢者にも理解しやすいシンプルな操作性を保つ
  - 「記録すること」自体が、水分補給を意識するきっかけになる
Should版
1. 就寝前の振り返りメールで達成状況を可視化する
  - 就寝前に振り返りメールを送信し、今日の記録(回数、合計水分量、達成率)を表示
  - 振り返りの機会を提供することで、ユーザー自身が「本当に飲んだか」を意識できる
2. 「後で飲む」機能と再リマインド
  - 記録時に「今すぐ飲んで記録」「後で飲む」を選択できる
  - 「今すぐ飲んで記録」：水分補給記録が1件追加される
  - 「後で飲む」：記録されず、30分後に再度リマインドメールが届く
  - この機能により、記録の信頼性が高まる
3. 連続達成日数のバッジ機能
  - 連続達成日数に応じてバッジを付与（例：7日連続達成で「1週間バッジ」）
  - 正直な記録を続けることで達成感を得られる仕組みを作る
4. 記録の信頼性を高める工夫
  - 記録時に「飲んだ時刻」を自動記録し、不自然な記録パターン（短時間に複数回記録など）を検知
  - 不自然なパターンが検出された場合、「本当に飲みましたか？」というメッセージを表示
  - ただし、ユーザーを疑うような表現は避け、あくまで「確認」として提示する
限界と割り切り：
- アプリ上で「実際に飲んだかどうか」を完全に検証することは不可能
- 最終的には、ユーザー自身の自主性に委ねる必要がある
- このアプリは「完全な監視」ではなく、「習慣化のサポート」を目的とする
- 「記録すること」自体が、水分補給を意識するきっかけになる
- 「カレンダーで達成状況を可視化する」ことで、継続のモチベーションを高める
- 「正直に記録することが、自分の健康のためになる」という意識を持ってもらうことが重要

他の人と関わる余地はある？

運用・仕組みで価値を出せないか？

機能以外の工夫：
- 定期的に「水分補給の豆知識」をメールで配信
- 連続達成日数に応じたバッジ機能（例：7日連続達成で「1週間バッジ」）
- 家族からの応援メッセージ機能
- まずは個人ユーザーでの実績を作り、その後、医療機関への展開を目指す


## 10. 技術スタック（手段としての技術）
### 10-1. 使用予定の技術
MVP版開発前の事前検証
- 認証だけの簡易アプリ作成
  - メールのリンクからサービスにスムーズに遷移できるかの検証
  - ログイン画面を経由する操作が、高齢者にとって負担になるかの検証

MVP版
- 言語：Ruby 3.2.3（または使用予定のバージョン）
- フレームワーク：Ruby on Rails 7.0.10
- DB：PostgreSQL
- デプロイ先：Render
- 使用予定ライブラリ：
  - Devise（ユーザー登録・ログイン機能）
  - Simple Calendar（カレンダー表示）
  - Action Mailer （メール通知機能）
  - Render Cron Jobs（定期実行）
  - Bootstrap または Tailwind CSS（デザイン）

Should版（MVP版完成後に追加）
- 使用予定ライブラリ：
  - Sidekiq（バックグラウンドジョブ）
  - Whenever（定期実行）
  - Action Mailer（メール通知機能）
  - トークン認証(Rails の has_secure_token)

Could版（拡張将来的に追加したい機能）
- 使用予定ライブラリ：
  - LINE Messaging API(LINE通知機能)

### 10-2. 技術選定の理由　
なぜこの技術を使うのか
| 技術 | 選定理由 |
| **MVP版** | |
| Ruby on Rails 7.0.10 | カリキュラムで学習済みで、安定したバージョン。MVCアーキテクチャで開発しやすい |
| PostgreSQL | Renderでのデプロイに適しており、本番環境での安定性が高い。開発環境から本番環境まで同じDBを使用することでトラブルを減らせる |
| Render | 無料枠でデプロイでき、初学者でも扱いやすい。PostgreSQLとの相性が良く、Cron Jobsで定期実行も簡単に設定できる |
| Devise | カリキュラムで学習済みで、多機能（パスワードリセット、メール認証など標準搭載）。初学者でも扱いやすい |
| Simple Calendar | カレンダー表示が簡単に実装でき、カスタマイズも容易。ビューのカスタマイズで達成状況の色分けが実現できる |
| Action Mailer | Railsの標準機能で、追加の学習コストが少ない。メール送信の基本機能を簡単に実装できる。|
| Render Cron Jobs | Renderのプラットフォーム機能で、シンプルに定期実行を設定できる。複雑な設定が不要で、MVP版に最適。|
| Bootstrap または Tailwind CSS | レスポンシブデザインが簡単に実装でき、高齢者向けの見やすいUIを作成できる |

| **Should版** | |
| Sidekiq | バックグラウンドジョブで非同期処理を実現し、定期的なメール送信に必要 |
| Whenever| cronを簡単に設定でき、指定時刻にメール送信を実行できる。Railsアプリケーション内でcronジョブを管理できるため、保守性が高い |
| Action Mailer | MVP版に引き続き使用。Should版では、Sidekiq + Wheneverと組み合わせてユーザーごとの通知時刻に対応 |
| トークン認証 | メールのリンクにトークンを含めることで、ログイン画面を経由せずに記録画面に直接遷移できる。高齢者のログイン負担を大幅に軽減できる。Railsの標準機（has_secure_token）で実装可能で、セキュリティも確保できる |
| SendGrid または Mailgun | メール配信の信頼性が高く、迷惑メールフォルダに振り分けられにくい。SPF、DKIM、DMARCの設定が簡単にでき、到達率を向上できる |

| **Could版** | |
| LINE Messaging API | LINEでの通知により、メールより確実にユーザーに届く。高齢者にも馴染みのあるツールで、使いやすい |

補足
※ MVP版では、Render Cron Jobsを使用して全ユーザーに一斉送信する仕組みを実装する。
　Should版では、Sidekiq + Wheneverを使用してユーザーごとの通知時刻に対応する。


不安な点
MVP版開発前の事前検証：
1. 認証だけの簡易アプリを作成できるか
  - Deviseの基本的な使い方は理解しているが、認証だけのシンプルなアプリを作成できるか不安
  - メールのリンクから簡易アプリに遷移する仕組みを実装できるか

MVP版：
1. Action Mailer + Render Cron Jobs の設定
不安な内容：
  - Action MailerとRender Cron Jobs を組み合わせた定期実行の設定が複雑
なぜ不安か：
  - カリキュラムで詳しく学習していないため、設定方法がわからない
対策：
  - 公式ドキュメントやQiita、Zennの記事を参考にする
  - わからないことがあれば、その都度質問する
  - シンプルな実装から始めて、段階的に理解を深める

2. Simple Calendar のカスタマイズ
不安な内容： 
  - カレンダーに達成状況を色分けして表示する点
なぜ不安か： 
  - Simple Calendarのカスタマイズ経験がない
対策：
  - 公式ドキュメントとサンプルコードを確認する
  - ビューファイルとCSSを工夫して、段階的に実装する
  - 高齢者にとってわかりやすいデザイン（大きな文字、はっきりした色）を意識する

Should版：
3. Sidekiq + Whenever の設定
不安な内容： 
  - Sidekiq と Whenever を組み合わせた定期実行の設定が複雑
なぜ不安か： 
  - カリキュラムで詳しく学習していないため、設定方法がわからない
対策：
  - MVP版では実装せず、Should版で取り組むことで時間的余裕を確保する
  - 公式ドキュメントやQiita、Zennの記事を参考にする
  - わからないことがあれば、その都度質問する

4. メール通知の到達率
不安な内容：
  - メールが迷惑メールフォルダに振り分けられる可能性がある
なぜ不安か：
  - メール配信の仕組みやSPF、DKIM、DMARCの設定経験がない
対策：
  - MVP版ではAction Mailer + Render Cron Jobsでシンプルなメール通知を実装する
  - Should版ではSendGridやMailgunなどの信頼性の高いメール配信サービスを使用する
  - 公式ドキュメントを参考にSPF、DKIM、DMARCを設定する
  - テスト送信を繰り返して、到達率を確認する
  - わからないことがあれば、その都度質問する

5. トークン認証の実装
不安な内容:
  - トークンを使った自動ログイン機能の実装方法がわからない
  - トークンの生成・検証・有効期限の設定方法がわからない
なぜ不安か:
  - カリキュラムで詳しく学習していないため、設定方法がわからない
  - セキュリティ面での考慮事項が不明である
対策:
  - MVP版では実装せず、Should版で取り組むことで時間的余裕を確保する
  - 公式ドキュメントやQiita、Zennの記事を参考にする
  - Rails の has_secure_token を使用してトークンを生成・検証する
  - わからないことがあれば、その都度質問する
  - シンプルな実装から始めて、段階的に理解を深める
