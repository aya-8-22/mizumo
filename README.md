# 名前：ミズモ（水＋メモ）

## 1.サービス概要
65歳以上の高齢者が、毎日の水分補給を習慣化するためのアプリです。
飲水記録を簡単に入力でき、カレンダーで達成状況を視覚的に確認できます。
メール通知機能により、飲水のタイミングを忘れずにリマインドします。

## 2.このアイデアはどこから生まれたか
### 2-1. きっかけとなった体験・感情
- いつ・どんな状況で：
医療機関で働いていた時、夏場になると発熱で搬送される高齢者が急増する状況を何度も目にしてきました。
特に印象的だったのは、高齢の患者様がよく「そういえば、あんまり水分とっていなかった。」と話していたことです。

- どんな気持ちになったか：
発熱があると、コロナウイルスやインフルエンザの可能性を考慮し、隔離や感染対策を行う必要があります。検査で陰性が確認されるまで、感染対策を継続しなければなりません。
その時、私は「もし水分補給を習慣化できていれば、この患者さんは熱中症にならず、医療機関の逼迫も予防できたのではないか」と強く感じました。
令和7年度の統計では、65歳以上の熱中症による救急搬送は全体の約6割（総務省 https://www.fdma.go.jp/disaster/heatstroke/items/r7/heatstroke_nenpou_r7.pdf）に及んでおり、この問題は個人の問題ではなく、医療現場全体の課題だと実感しました。

### 2-2. なぜそれが「気になった」のか
- 自分の性格、過去の経験
医療現場で働いていた経験から、「予防できる病気で医療機関が逼迫する」という状況に強い違和感を感じていました。
高齢者は体温調節機能の低下、認知機能や判断力の低下により、水分摂取を忘れやすい傾向にあります。しかし、「水分補給を忘れないようにする」という習慣化さえできれば、熱中症は予防できるはずです。

- 大切にしている価値観：
私は「予防できる病気は予防すべき」という考えを大切にしています。
特にコロナ禍以降、医療機関では感染対策の負担が増しており、「予防できる患者を減らすことで、医療現場の負担を軽減したい」という思いが強くありました。
また、医療現場で働いていた時に、「もっと早く対策できていれば」と感じる場面を何度も経験したことが、このアプリを作りたいと思った理由です。


3. 課題の整理（表面的になっていないか）
3-1. 表に見えている困りごと
- 高齢者が水分補給を忘れやすい傾向にあることです。

3-2. 本当に解決したい課題は何か
- なぜ水分補給を忘れるのか？
  - 喉の渇きを感じにくくなっているからです。

- なぜ喉の渇きを感じにくいのか？
  - 加齢により、体の水分量が減少し、喉の渇きを感じる機能が低下しているからです。

- なぜそれが問題なのか？
  - 水分不足が続くと、脱水症状や熱中症のリスクが高まり、健康を損なうからです。

- 本当に解決したい課題：
  -高齢者が、喉の渇きを感じなくても、定期的に水分補給を思い出し、習慣化できるようにすること。

## 4. 想定ユーザーについて
### 4-1. 想定しているユーザー
年齢設定：
- 年齢：
65歳〜75歳の比較的元気な高齢者。
- 理由：
熱中症による救急搬送の6割が65歳以上であり、この年代にアプローチすることで、熱中症予防に大きな効果が期待できます。また、65歳〜75歳は スマホやメールを日常的に使っている方が多いため、アプリの導入ハードルが低いと考えました。

生活・状況設定：
- 生活・状況：
今回は飲水制限がなく、日常生活を自宅で送っている高齢者を対象とする。

- 理由：
他者を介さずに、アプリの力で、定期的な飲水摂取を促すことができるため。

環境設定：
- 環境：
メール通知を採用します。
- 理由：
高齢者の中には、LINEやスマホアプリに不慣れな方も多くいます。一方で、メールは ガラケー時代から使い慣れている方が多く、スマホに機種変更した後もメールアプリは標準でインストールされているため、導入のハードルが低いと考えました。ただし、将来的には LINE通知にも対応することで、より幅広く使ってもらえるようにしたいと考えています。

将来的な展開：
- まずは元気な高齢者で実績を作り、その後 医療機関や飲水制限のある高齢者にも展開したいと考えています。
- 医療機関や介護施設での導入には、医師の監修や飲水制限への対応が必要になるため、まずは個人ユーザーでの実績を作ることを優先します。

## 8.このアプリの懸念点とその対策
懸念点①：メール通知が迷惑メールフォルダに入ってしまう
何が問題になりそうか：
 - メール通知が迷惑メールフォルダに振り分けられると、ユーザーがリマインドを受け取れない
なぜそう思うか：
 - 定期的に送信されるメールは、スパムと判定されやすい
そのために考えている対策：
MVP
 - ユーザーがメールアドレスを登録する際に、テストメールを送信して受信確認を促す
 - 初回登録時に「メールが届かない場合は迷惑メールフォルダを確認してください」と案内
本リリース
 - 送信元ドメインの認証（SPF、DKIM）を設定して信頼性を高める

---

懸念点②：ユーザーが飲水記録を入力し忘れる
何が問題になりそうか：
 - 飲水記録を入力しないと、達成状況が正しく反映されず、アプリの価値が下がる
なぜそう思うか：
 - 高齢者は、スマホの操作に慣れていないため、記録を忘れやすい
そのために考えている対策：
 - 記録入力を簡単にする（ワンタップで記録完了）
 - メール通知に「記録はこちらから」というリンクを含める
 - 就寝時間前に「今日の記録を確認しましょう」というリマインドメールを送る
MVP:
 - 飲水量：コップ１杯200mlとする
本リリース
 - 設定画面で「コップ1杯の量」を選択できるようにする
 - 選択肢：150ml / 200ml / 250ml など
 - 記録時は、設定した量が自動的に記録される

---

懸念点③：目標水分量の計算が実態と合わない可能性
何が問題になりそうか：
 - 体重だけで目標水分量を計算すると、個人の生活習慣や健康状態に合わない場合がある
なぜそう思うか：
 - 運動量、気温、持病などによって必要な水分量は変わるため
そのために考えている対策：
 - MVP版では「体重 × 30ml」というシンプルな計算式を採用
 - 本リリース版では、運動量や気温を考慮したカスタマイズ機能を追加
 - 「医師に相談してください」という注意書きを表示

今回チャレンジしたい点
MVP版でチャレンジすること：
Simple Calendar のカスタマイズ
 - カレンダーに達成状況を色分けして表示する
 - ユーザーごとの記録をカレンダーに反映させる
 - ビューとCSSを工夫して、視覚的にわかりやすいUIを実現
高齢者向けのユーザー体験の最適化
 - 大きな文字サイズとはっきりした色のコントラスト
 - シンプルで直感的なレイアウト
 - アイコン（✅、❌など）を活用した視覚的なフィードバック
PostgreSQL を使用した開発
 - 開発環境から本番環境まで同じデータベースを使用する経験
 - 本番環境を意識したデータベース設計

Should版でチャレンジすること：
Sidekiq + Whenever の組み合わせ
 - 定期的なメール通知を実装する経験
 - バックグラウンドジョブの仕組みを理解し、非同期処理を実装
メール通知機能の実装
 - ユーザーが設定した時刻にメール通知を送信
 - メールの文面やタイミングを工夫して、ユーザーが確実に受け取れるようにする
メール到達率の向上
 - SendGrid や Mailgun などのメール配信サービスを導入
 - SPF、DKIM、DMARCの設定を行い、迷惑メールフォルダに振り分けられないようにする

不安な点
1. Sidekiq + Whenever の設定（Should版）
不安な内容： Sidekiq と Whenever を組み合わせた定期実行の設定が複雑そう
なぜ不安か： カリキュラムで詳しく学習していないため、設定方法がわからない
対策：
MVP版では実装せず、Should版で取り組むことで時間的余裕を確保
公式ドキュメントやQiita、Zennの記事を参考にする
わからないことがあれば、その都度質問する

2. Simple Calendar のカスタマイズ
不安な内容： カレンダーに達成状況を色分けして表示する方法
なぜ不安か： Simple Calendar のカスタマイズ経験がない
対策：
公式ドキュメントとサンプルコードを確認
ビューファイルとCSSを工夫して、段階的に実装
高齢者にとってわかりやすいデザインを意識（大きな文字、はっきりした色）

3. メール通知の到達率（Should版）
不安な内容： メールが迷惑メールフォルダに振り分けられる可能性
なぜ不安か： メール配信の仕組みやSPF、DKIM、DMARCの設定経験がない
対策：
MVP版ではメール通知機能は実装せず、Should版で取り組む
SendGrid や Mailgun などの信頼性の高いメール配信サービスを使用
公式ドキュメントを参考にSPF、DKIM、DMARCを設定
テスト送信を繰り返して、到達率を確認
